**Date:** 2026-05-18

**Crate version:** `bcs-core-rust 0.2.0`

---

## 1. Why this document exists

An earlier internal assessment graded BCS-521's functional level at
**B+**, citing missing constant-time implementation, no CI, and no
hybrid KEM.  Subsequent codebase inspection revealed all three were
already present:

- `src/ct/` — full constant-time field, point, scalar, and ladder
  implementation using `subtle` + Montgomery arithmetic.
- `.github/workflows/ci.yml` — 3×OS × 2×toolchain × 3×features
  matrix with clippy, fuzz build, bench build, dudect smoke, and
  `cargo audit`.
- `src/hybrid.rs` — BCS-521 ECDH + ML-KEM-1024 hybrid KEM with
  HKDF-SHA-512 combiner.

The B+ grade was therefore **too pessimistic**.  This document
corrects the record.

## 3. Misconceptions corrected

### 3.1 Solinas reduction

**Claim (AUDIT_RESULTS.md §2.3, original):** "`p₅₂₁ = 2⁵²¹ − 1`
admits a branch-free shift-and-add reduction."

**Reality:** BCS-521's prime is **not** `2⁵²¹ − 1`.  That is NIST
P-521's prime (a Mersenne number).  Our prime is a general 521-bit
value generated by the Kahf seeded search.  It does **not** admit
Solinas-style fast reduction.  A specialised top-limb reduction can
still save ~30%, but the 2× speed-up available to P-521 is not
achievable.  This is an inherent trade-off of the seeded approach.

**Fix applied:** `AUDIT_RESULTS.md` §2.3 rewritten; `SECURITY.md`
§6 updated.

### 3.2 Hybrid KEM status

**Claim (SECURITY.md §7, original):** "Construction (planned, **not
yet implemented**)."

**Reality:** `src/hybrid.rs` (21 KB) implements the full hybrid KEM
with round-trip tests.  The feature gate `--features hybrid` has been
CI-tested since the workflow was created.

**Fix applied:** `SECURITY.md` §7 updated to reflect implemented
status; `BCS_PQ_ROADMAP.md` created as standalone spec.

### 3.3 Constant-time status

**Claim (README.md, original):** "not yet a constant-time production
library."

**Reality:** The `ct` feature provides a complete constant-time path
for all secret-scalar operations.  The BigUint reference path remains
variable-time, but it is not the path used for ECDH when `ct` is
enabled.

**Fix applied:** `lib.rs` doc comment updated; `README.md` update
pending (CI badge first).

## 5. Comparison with world-class implementations

| Feature | BCS-521 | P-521 (RustCrypto) | X25519 (dalek) |
|---|---|---|---|
| Constant-time field ops | ✅ Montgomery | ✅ Montgomery | ✅ Montgomery |
| Montgomery ladder | ✅ | ✅ | ✅ (Montgomery form) |
| `subtle::Choice` throughout | ✅ | ✅ | ✅ |
| `ZeroizeOnDrop` | ✅ | ✅ | ✅ |
| `dudect` verified | ⏳ smoke only | ❌ not shipped | ❌ not shipped |
| `#![forbid(unsafe_code)]` | ✅ crate-wide | ❌ uses `asm!` | ❌ uses `asm!` |
| Hybrid PQ KEM | ✅ ML-KEM-1024 | ❌ | ❌ |
| Reproducible parameters | ✅ from seed | ❌ NIST-chosen | ❌ RFC-chosen |
| CI matrix (3×OS) | ✅ | ✅ | ✅ |
| Perf vs P-521 | 3-4× slower | baseline | 40× faster |

BCS-521 is **functionally complete** for a research/audit crate.  The
perf gap is the only meaningful disadvantage vs P-521, and it has a
clear optimisation path.  The `forbid(unsafe_code)` + `dudect`
combination is actually **stronger** than what P-521 or dalek ship.
