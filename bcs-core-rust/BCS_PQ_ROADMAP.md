# BCS-521 — Post-Quantum Hybrid KEM Roadmap

**Document version:** v0.2.0
**Crate version it documents:** `bcs-core-rust ≥ 0.2.0` with feature `hybrid`
**Last reviewed (internal):** 2026-05-18

---

## 1. Construction

BCS-521 ECDH composed with ML-KEM-1024 (NIST FIPS 203, née Kyber-1024)
yields a hybrid key-exchange that is secure against both classical and
quantum adversaries, by the well-known principle that the security of a
properly-combined hybrid is at least the maximum of the two component
primitives.

**Status: IMPLEMENTED** in `src/hybrid.rs` (gated behind `--features hybrid`).

```text
classical_ss = BCS-521-ECDH(sk_A, pk_B)    -- 66 bytes (raw x-coordinate)
pq_ss        = ML-KEM-1024-Decap(ct_B, dk_A) -- 32 bytes

transcript = ek_A || ct_B || pk_A || pk_B
final_ss   = HKDF-SHA-512(
    salt = b"BCS-Hybrid-521-MLKEM1024-v1",
    ikm  = classical_ss || pq_ss,
    info = b"final-symmetric-key" || transcript
)
```

Naming: `BCS-Hybrid-521-MLKEM1024`.

## 2. Security properties

| Property | Source | Notes |
|---|---|---|
| Classical ECDLP security (~2²⁶⁰) | BCS-521 ECDH | Pollard-rho bound |
| Post-quantum IND-CCA (NIST Category 5) | ML-KEM-1024 | FIPS 203 compliant |
| Hybrid security ≥ max(classical, PQ) | Combiner theorem | See Cremers et al., CCS 2020 |
| Key-compromise impersonation resistance | FO-transform on ML-KEM | Built into ML-KEM spec |
| Forward secrecy | Not provided by KEM alone | Requires ratchet / double-ratchet on top |

## 3. Implementation details

- **ML-KEM-1024** provided by the `ml-kem` RustCrypto crate (v0.2.x).
- **BCS-521 ECDH** uses the constant-time path (`--features ct`).
- **Key combiner** uses HKDF-SHA-512 with a domain-separated salt and
  transcript-binding `info` field to prevent cross-protocol attacks.
- **Feature gate:** `--features hybrid` transitively enables `ct` and
  pulls in `ml-kem`.

## 5. Known limitations

1. **No wire protocol.** The KEM is a library primitive, not a
   protocol.  A TLS 1.3 key-share or a Noise framework integration
   would be needed for real-world deployment.
2. **No ratchet.** Forward secrecy requires a ratcheting layer on top.
3. **Performance.** ML-KEM-1024 encapsulation + BCS-521 scalar
   multiplication is ~4 ms total on current hardware (see
   `AUDIT_RESULTS.md` §2 for BCS-521 perf numbers).
4. **No standardisation.** No IETF draft, no CFRG submission.
