# BCS-521 — Threat Model

**Document version:** v0.2.0
**Crate version it documents:** `bcs-core-rust ≥ 0.2.0` with feature `ct`
**Last reviewed (internal):** 2026-05-18

---

## 1. Assets

| Asset | Sensitivity | Location |
|---|---|---|
| ECDH secret scalar `s` | **Critical** — knowledge breaks ECDH | `Bcs521SecretKey` (zeroed on drop) |
| ECDH shared secret `x(s·P)` | **Critical** — the session key precursor | `Bcs521SharedSecret` (zeroed on drop) |
| ML-KEM decapsulation key `dk` | **Critical** — knowledge breaks PQ KEM | `MlKemDecapKey` (owned by `ml-kem` crate) |
| Hybrid final key | **Critical** — the actual session key | Output of HKDF-SHA-512 |
| ECDH public key `s·G` | **Public** | `Bcs521PublicKey` |
| ML-KEM encapsulation key `ek` | **Public** | `MlKemEncapKey` |
| Curve parameters (`p`, `n`, `G`, `a`, `b`) | **Public** | `src/ct/consts.rs` |
| Kahf master seed | **Public** — deterministic, not secret | `src/kahf_seeded.rs` |

## 2. Attacker profile

### In scope

| Attacker | Capability | Motivation |
|---|---|---|
| **Network passive** | Observes ciphertexts, public keys | Session key recovery |
| **Network active (MITM)** | Modifies messages in transit | Key substitution, impersonation |
| **Local process** | Reads process memory, `/proc/pid/mem` | Secret key extraction |
| **Timing analyst** | Measures operation latency with nanosecond precision | Side-channel key recovery |
| **Cache attacker** | Measures shared cache state (L1/L2) | Cache-timing key recovery |
| **Quantum adversary** | Possesses a large-scale fault-tolerant QC | Shor's algorithm on ECDLP |

### Out of scope

| Attacker | Why out of scope |
|---|---|
| **Physical attacker** (power analysis, EM probe, fault injection) | Requires hardware countermeasures beyond software scope |
| **OS kernel / hypervisor** | If the kernel is compromised, no software defence applies |
| **Supply-chain attacker** (compromised Rust compiler, crate registry) | Addressed by reproducible builds / cargo audit, not by this crate |
| **Microarchitectural attacker** (Spectre, Meltdown, Foreshadow) | OS/hardware responsibility; software mitigations are incomplete |

## 3. Threats and mitigations

### 3.1 Classical cryptographic attacks

| Threat | Mitigation | Status |
|---|---|---|
| **Pollard-rho ECDLP** | Group order ≈ 2⁵²¹ → generic DL ≈ 2²⁶⁰ | ✅ structural |
| **Small-subgroup / invalid-curve** | Mandatory on-curve + range + non-identity check in `from_bytes` | ✅ enforced |
| **Twist attack** | Composite twist order; mitigated by rejecting off-curve points | ✅ enforced |
| **MOV / Frey–Rück pairing** | Embedding degree too large for sub-exponential attack | ✅ verified |
| **Smart's anomalous attack** | `#E(F_p) ≠ p` | ✅ verified |
| **Identity-output attack** | `ecdh` returns error rather than degenerate key | ✅ enforced |

### 3.2 Implementation-level attacks

| Threat | Mitigation | Status |
|---|---|---|
| **Timing side-channel (secret scalar)** | Montgomery ladder + `subtle::Choice` + always-mul in `invert` | ✅ designed + smoke-verified |
| **Timing side-channel (long-budget)** | `dudect` harness exists; long run on baremetal pending | ⏳ pending |
| **Memory disclosure of secrets** | `ZeroizeOnDrop` + redacted `Debug` | ✅ enforced |
| **Invalid-curve point injection** | `from_bytes` validates before any scalar mul | ✅ enforced |
| **Key-compromise impersonation** | FO-transform on ML-KEM; transcript-bound HKDF | ✅ enforced |
| **Cross-protocol attack** | Domain-separated HKDF salt + transcript in `info` | ✅ enforced |

### 3.3 Post-quantum attacks

| Threat | Mitigation | Status |
|---|---|---|
| **Shor's algorithm on ECDLP** | ML-KEM-1024 hybrid component (NIST Cat 5) | ✅ via hybrid KEM |
| **Grover's search on AES-256** | 256-bit final key from HKDF-SHA-512 | ✅ 2¹²⁸ post-quantum security |

### 3.4 Supply-chain / operational threats

| Threat | Mitigation | Status |
|---|---|---|
| **Backdoor in curve parameters** | Fully reproducible from public seed; auditor recomputes in seconds | ✅ enforced |
| **Dependency vulnerability** | `cargo audit` in CI; pinned minor versions for crypto deps | ⏳ CI advisory |
| **Compiler-introduced timing leak** | `opt-level=3` + `dudect` empirical test | ⏳ LLVM-IR inspection is auditor responsibility |

## 4. Assumptions

1. **ECDLP hardness.** Generic discrete log in `<G> ⊂ E(F_p)` requires
   ≈ √n ≈ 2²⁶⁰ group operations. Standard assumption shared by all
   production curves.
2. **No special algebraic structure.** The curve was generated by a
   parameter-search with no designer-chosen freedom (a, b, c fixed;
   only p searched). No backdoor in the Dual-EC sense.
3. **Random-oracle model for HKDF.** Standard assumption for HKDF-based
   key derivation.
4. **ML-KEM-1024 IND-CCA security.** Assumed per NIST FIPS 203
   evaluation. The `ml-kem` RustCrypto crate is the implementation.
5. **Trusted execution environment.** The OS is not compromised; the
   process runs on unvirtualised hardware (or a trusted VM).

## 5. Residual risk

Even with all mitigations in place, the following residual risks remain:

1. **Compiler-introduced side channels.** LLVM may introduce
   data-dependent branches during optimisation. Only `dudect` on the
   actual deployed binary can detect this.
2. **Microarchitectural attacks.** Spectre-class attacks are not
   mitigated in software.
3. **No formal verification.** The constant-time claim is empirical,
   not proven.
4. **No external audit.** No third party has reviewed the code.
