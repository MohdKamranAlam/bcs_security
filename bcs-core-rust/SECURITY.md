> **TL;DR for the impatient reader.**
> BCS-521 is a 521-bit prime-field Weierstrass curve with a verified
> independent cardinality proof, a constant-time Rust implementation,
> and 90+ passing tests including byte-exact parity against a BigUint
> reference.  It is **suitable for academic publication, internal R&D,
> and demonstration deployments today.**  It is **not yet suitable for
> production systems that protect data of consequence** ‚ÄĒ production
> use is gated on (a) an external cryptographic audit, (b) the
> `dudect` side-channel statistical test, and (c) a public-comment
> period of at least 90 days.  Read on for the precise list of
> guarantees, assumptions, and out-of-scope concerns.

## 2. What is *proven*

A claim is in this section only if either (a) we have a mechanised
verification artefact, or (b) the claim follows from an independently
audited library.

### 2.1 Prime fields

* **`p` is prime.**  Verified by Miller-Rabin (40 rounds) + Pari/GP
  `isprime`.  No counterexample known and not expected; for a 521-bit
  random prime this is overwhelming.

* **`n` is prime.**  Verified by Pari/GP `isprime` (deterministic for
  this size on modern Pari) on the output of SEA point counting.

### 2.2 Elliptic-curve structure

* **`#E(F_p) = n` exactly.**  Computed by Schoof-Elkies-Atkin in
  Pari/GP.  Run log archived as `bcs-verify/bcs521_sage_proof.sage`
  output.  Total runtime ‚Čą 10 minutes on a 4-core Codespaces machine.

* **`G` has order `n`.**  Verified by `n ¬∑ G = O` (test
  `bcs521_n_times_generator_is_infinity` in `src/lib.rs`).

* **`(n‚ąí1)¬∑G + G = O`.**  Sanity check that confirms order is *exactly*
  `n`, not a proper divisor (test
  `bcs521_n_minus_one_plus_one_is_infinity`).

### 2.3 Implementation correctness

* **Reference BigUint implementation:** passes 10/10 frozen test
  vectors (`tests/test_vectors_521.rs`) covering keygen, scalar mul,
  ECDH, HKDF-SHA-512 derivation, Kahf-domain-separator binding, and
  AES-256-GCM round-trip.

* **Constant-time implementation:** passes 9/9 byte-exact parity
  tests against the BigUint reference for scalars
  `k ‚ąą {1, 2, 3, 7, 42, 2141, 2969, u64::MAX, 256-bit random}`
  (`tests/test_ct_parity.rs`).  This validates the entire CT stack:
  9-limb CIOS Montgomery multiplication, Fermat inversion,
  Renes‚ÄďCostello‚ÄďBatina complete addition and doubling, and the
  Montgomery ladder.

## 3. What is *assumed*

A claim is in this section only if it is a standard hardness assumption
or a property believed (but not proven from first principles) to hold
for any well-chosen curve.

* **ECDLP hardness.**  Solving the discrete-log problem in
  `<G> ‚äā E(F_p)` is assumed to require effort `‚Čą ‚ąön ‚Čą 2^260` group
  operations using the best known generic algorithm (Pollard rho).
  This is the same assumption made by P-256, P-384, P-521, secp256k1,
  Curve25519, and every other production curve in use.

* **No special algebraic structure.**  The curve is generated by the
  parameter-search procedure in `bcs-spec/bcs-521.md` with no
  designer-chosen freedom (a = ‚ąí2, b = 5, c = 4 are fixed from the
  *Bismillah Diophantine* equation in the project's mathematical
  foundation; only the prime `p` was searched).  We therefore believe
  the curve carries no backdoor in the sense of NSA-Dual-EC.

* **Random-oracle for HKDF-SHA-256.**  The KDF used to derive
  symmetric secrets from the raw shared x-coordinate is modelled as
  a random oracle, the standard practice for HKDF.

## 4. Attack mitigations

| Attack | Mitigation | Status |
|---|---|---|
| **Small-subgroup / invalid-curve** | Mandatory point validation in `Bcs521PublicKey::from_bytes` (on-curve check + coord-range check + non-identity check). | ‚úÖ enforced |
| **Twist attack** | Twist order is composite ‚Üí invalid-curve attack possible *if* a peer accepts off-curve points.  **`Bcs521PublicKey::from_bytes` rejects all off-curve points before any scalar multiplication.** | ‚úÖ enforced |
| **Identity-output attack** | `Bcs521::ecdh` returns `EcdhResultIsIdentity` rather than a degenerate symmetric key in the impossible case of `s¬∑peer_pk = O`. | ‚úÖ enforced |
| **Timing side-channel (secret scalar)** | All scalar-secret operations route through the Montgomery ladder over Renes‚ÄďCostello‚ÄďBatina formulas, with `subtle::ConditionallySelectable` for branchless selection.  An empirical `dudect` (Reparaz 2017) timing-leak harness ships in `examples/dudect_ct.rs` covering `Fp521::mont_mul`, `scalar_mul_generator`, and end-to-end `Bcs521::ecdh`.  **First public smoke run (2026-05-17, Codespaces): `\|t\| < 2.3` on all three benches, well below the audit threshold of `\|t\| < 4.5`** ‚ÄĒ see [`AUDIT_RESULTS.md`](AUDIT_RESULTS.md) ¬ß1. Long-budget run on a quiescent baremetal machine is the next milestone. | ÔŅĹ designed + smoke-verified; long-budget run pending |
| **Memory disclosure of secrets** | `Bcs521SecretKey` and `Bcs521SharedSecret` implement `zeroize::ZeroizeOnDrop`.  `Debug` impls deliberately redact: printing yields `"<redacted>"`. | ‚úÖ enforced |
| **Pollard rho ECDLP** | Group order ‚Čą `2^521`, so generic discrete log ‚Čą `2^260`.  Comfortably above the current 80-bit ‚Äúquasi-broken‚ÄĚ bar and on par with P-521. | ‚úÖ structural |
| **MOV / Frey-R√ľck embedding** | Embedding degree of `E[n]` into `F_{p^k}` was computed and is too large for any sub-exponential attack to apply.  See `BCS_SECURITY_AUDIT_COLAB.py`. | ‚úÖ verified |
| **Anomalous (Smart) attack** | `#E(F_p) ‚Č† p`.  Verified directly. | ‚úÖ verified |
| **Quantum (Shor's algorithm)** | **Not mitigated.**  Like every classical ECC, BCS-521 falls to Shor's algorithm on a sufficiently large fault-tolerant quantum computer.  See ¬ß7 for the hybrid post-quantum roadmap. | ‚ĚĆ out of scope |
| **Fault injection / side channels other than timing** (EM, power, cache) | **Not addressed** in software; would require hardware countermeasures. | ‚ĚĆ out of scope |

## 5. What is **not yet** audited or verified

This section is updated honestly with every release.  Anything below
this line is a known gap, *not* a known vulnerability.

* **No external cryptographic audit.**  The implementation has not been
  reviewed by NCC Group, Cure53, Trail of Bits, or any equivalent
  third party.  Internal review only.

* **No long-budget `dudect` run on a quiescent machine.**  The CT
  code is *designed* to be constant-time, and the algorithms are
  textbook constant-time.  But the LLVM compiler can in principle
  introduce data-dependent branches via optimisation.  A `dudect`
  harness exists at `examples/dudect_ct.rs` covering three benches:
  `bcs521_scalar_mul`, `bcs521_ecdh`, and `fp521_mont_mul`, each as
  a fixed-vs-random Welch t-test.  A passing verdict requires
  `|t| < 4.5` (‚Č° `p > 10‚ĀĽ‚ĀĶ`) sustained over ‚Č• 10‚Ā∂ measurements on a
  quiescent baremetal machine ‚ÄĒ that long-budget run is the next
  audit milestone.

  **First public smoke run** (2026-05-17, GitHub Codespaces, 5 000‚Äď
  106 000 samples per bench): max `|t|` ‚ąą {1.84, 2.21, 2.28}, all
  three benches clear the `|t| < 4.5` bar. See
  [`AUDIT_RESULTS.md`](AUDIT_RESULTS.md) ¬ß1 for the full table,
  caveats, and reproducer.

  **CI smoke run** (`fp521_mont_mul`, single round) is wired into
  `.github/workflows/ci.yml` to catch catastrophic regressions per
  PR; it is **advisory only** until a baseline t-distribution is
  established.

* **No long-budget fuzz run.**  `cargo-fuzz` targets ship in `fuzz/`
  for `Bcs521PublicKey::from_bytes`, `Bcs521SecretKey::from_bytes`,
  and an ECDH round-trip target.  They build on every CI run, but
  no continuous run of `‚Č• 1 hour` per target has yet been published
  with corpus + crash-set.

* **No formal verification of the CT code.**  Tools like HACL\*,
  Fiat-Crypto, or EverCrypt have not been applied.

* **No published baseline performance numbers on reference hardware.**
  A multi-curve Criterion bench harness ships at
  `benches/ecdh_compare.rs` comparing BCS-521 against `p256`, `p521`,
  `k256`, and `x25519-dalek` for both keygen and ECDH.  The fair
  cousin-to-cousin comparison is **BCS-521 vs P-521**.

  **First public smoke run** (2026-05-17, GitHub Codespaces) headline:
  BCS-521 keygen ‚Čą 2.93 ms, ECDH ‚Čą 3.10 ms; P-521 keygen ‚Čą 918 ¬Ķs,
  ECDH ‚Čą 759 ¬Ķs (i.e. BCS-521 is **3‚Äď4√ó slower than P-521** in this
  configuration).  The gap is attributed to three known
  un-applied optimisations (specialised top-limb reduction, fixed-base
  comb, mixed-add specialisation), all of which preserve the CT
  contract.  Note: unlike NIST P-521, BCS-521's prime is not a
  Mersenne number and does not admit Solinas-style fast reduction;
  see `AUDIT_RESULTS.md` ¬ß2.3 for the honest analysis.
  See [`AUDIT_RESULTS.md`](AUDIT_RESULTS.md) ¬ß2 for the full table
  and reproducer.  A re-run on dedicated baremetal hardware is the
  next step.

* **No standardisation track.**  No IETF draft, no CFRG submission,
  no NIST SP 800-186 inclusion request.

* **The hardcoded 2G test vector** in `tests::bcs256_small_scalar_vector`
  is `#[ignore]`'d pending an independent Sage computation.  This is a
  test-hygiene gap, not a correctness gap (the underlying arithmetic
  passes `2G_via_scalar_mul == G + G`).

**Production use of any code depending on this crate should be gated on
at least the first three items in this list.**

## 6. Out of scope

We explicitly do **not** claim or attempt the following:

1. **Replacing P-256 / P-384 / P-521 / Curve25519 in deployed systems.**
   This requires (a) external audit, (b) standardisation, (c) wide
   adoption by independent implementers ‚ÄĒ a multi-year process for
   *any* new curve.  We are at year zero.

2. **Compliance with FIPS 140-3 or Common Criteria.**  No certification
   has been pursued.

3. **Hardware-side-channel resistance.**  Power analysis, EM analysis,
   acoustic, and microarchitectural attacks (Spectre, Meltdown, etc.)
   are not in the threat model addressed by this software.

4. **Quantum-secure key exchange standalone.**  See ¬ß7 for the
   implemented hybrid construction.

## 7. Post-quantum roadmap

**The plan:** BCS-521 ECDH composed with ML-KEM-1024 (NIST FIPS 203,
n√©e Kyber-1024) yields a hybrid key-exchange that is secure against
both classical and quantum adversaries, by the well-known principle
that the security of a properly-combined hybrid is at least the
maximum of the two component primitives.

Construction (**implemented** in `src/hybrid.rs`, gated behind
`--features hybrid`):

```text
classical_ss = BCS-521-ECDH(sk_A, pk_B)    -- 66 bytes (raw x-coordinate)
pq_ss        = ML-KEM-1024-Decap(ct_B, dk_A) -- 32 bytes

transcript = ek_A || ct_B || pk_A || pk_B
final_ss   = HKDF-SHA-512(
    salt = b"BCS-Hybrid-521-MLKEM1024-v1",
    ikm  = classical_ss || pq_ss,
    info = b"final-symmetric-key" || transcript
)
```

Naming: `BCS-Hybrid-521-MLKEM1024`.

See `BCS_PQ_ROADMAP.md` for the full specification, security
properties, and future work.

## 8. Reporting a security issue

If you believe you have found a security issue:

* **Do not** open a public GitHub issue.
* Email `bcs-security@[domain-to-be-published]` with a clear
  reproduction, expected versus actual behaviour, and (ideally) a
  proof-of-concept patch or test vector.
* Allow 90 days for coordinated disclosure before publishing.

We commit to acknowledging receipt within 72 hours and to publishing
a remediated release within 90 days for any confirmed issue rated
"high" or above.
