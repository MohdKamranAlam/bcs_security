## Twist Analysis

### Factorization

```text
#E_twist = 43 × 47 × 50551 × R
R = 495-bit composite (independent Python verification, sympy BPSW)
```

The remaining 495-bit factor R is composite (not PRP). This is **not a security failure** — the twist has no exploitable small-subgroup structure because:
1. All small prime factors (43, 47, 50551) are tiny — easily checked and rejected by point validation.
2. The 495-bit remaining factor is far too large for any practical twist attack.
3. **Mandatory point validation** prevents any twist attack regardless of factorization.

### Verdict on Twist

```text
BCS-521: STRONG CLASSICAL ECC PASS
         with mandatory point validation

Composite twist is NOT a weakness — Curve25519 also has composite twist
and relies on point validation (via Montgomery ladder). Same principle applies.
```

## Twist Policy

Twist order `n' = 2(p+1) − n` is composite. Per BCS Twist Validation Policy:

- **Mandatory point validation** before every operation.
- Reject any point not satisfying the curve equation.
- Reject any point with small-subgroup order (cofactor check: `h·P ≠ O` check).
- Implementations MUST validate public keys.

BCS-521 supports an **optional Kahf binding** that cryptographically anchors
protocol outputs to the Surah Kahf prime structure (Quran 18:1 - 18:110),
following the BIP-340 / RFC 9180 HPKE pattern for domain separation.

### Canonical Encoding

```text
canonical_input(label) = label ":" ( key "=" decimal(value) ";" ){5 in alphabetical key order}
DST(label)             = SHA-256( canonical_input(label) )    // 32 bytes
```

For `label = "BCS-521-Kahf-v1"`, the canonical input is exactly:


### Kahf-Bound HKDF

The Kahf DST is prepended to the HKDF `info` field:

