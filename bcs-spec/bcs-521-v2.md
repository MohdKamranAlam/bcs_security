# BCS-521-V2 Curve Specification (DRAFT)

## Status

**DRAFT — V2 cipher suite of the BCS-521 family.**

This spec proposes a *deterministically derived* prime `p` whose origin is
provable from the canonical Kahf seed input. It is a **separate cipher
suite** from the existing **BCS-521 (V1)** in `bcs-spec/bcs-521.md`.

**V1 is FROZEN and untouched.** Both V1 and V2 will coexist as independent
suites; deployments may select either.

The full search (~hours on multi-core) must be run before this spec can
be frozen with concrete `p` and `n` values.

## 2. Motivation — Trapdoor-Free Origin

V1's prime was found by random search. It satisfies every audit
criterion, but its *origin* is unexplained — a verifier has to take it on
faith that no trapdoor was inserted in the seed of that random search.

V2 mirrors the **NIST P-256 verifiably-random** methodology
(FIPS 186-4 Appendix A.1.2), but uses Quranic Kahf-prime constants + the
Bismillah master equation as the seed material instead of an unexplained
160-bit hex string. Anyone holding the seed text can:

1. Re-derive every candidate `p` byte-for-byte, in order.
2. Confirm that the first counter producing a valid `(p, n)` pair is the
   published winner — no skipping, no cherry-picking.
3. Verify that no portion of `p` was chosen freely by the constructor.

This collapses "trust the constructor" into "trust SHA-512 + the Quranic
constants", which is the strongest provenance argument possible for a
fresh elliptic curve.

V1 still gives the same `(p, n)` quality; V2 gives the same quality
*plus* a verifiable origin.

## 3. Curve

Same Bismillah curve as V1:

```text
E: y² = x³ − 2x² + 5x + 4   over   F_p
```

Origin: master equation `T_A = 17·B² + 5·B + 4 = 6236` with `B = 19`
(letters of Bismillah).

Generator (independent of `p`, since `4 ≡ 0³ − 2·0² + 5·0 + 4`):

```text
G = (0, 2)
```

```text
                =  a7e2095812a53b18111510409951b3472dcdbfdc49a08600
                   dd83f3b644a8ebeddcd856198544a56d905272203057ee7b
                   6c1a55b080fd8d51a9144b739ed95cbd   (64 bytes)
```

The label `BCS-521-V2-Seed-v1` is structured as:
- `BCS-521-V2` — the cipher-suite generation (V2 of BCS-521).
- `-Seed-v1` — the seed-encoding schema version. Bump to `v2` *only* if
  the canonical-input format itself ever changes.

The 5 Kahf primes are listed in **alphabetical key order**, matching the
Kahf Domain Separator binding in `bcs-521.md`. Curve coefficients
(`a2, a4, a6`) and `bits` are appended so any change to the curve form or
target bit length yields a different prime stream.

### What changes invalidate the seed?

Any of the following → *different* master seed → different candidate stream:

- Label string (e.g., `BCS-521-V2-Seed-v2`, or `BCS-521-V3-Seed-v1`)
- Any Kahf prime value
- Any curve coefficient
- Target bit length
- Order of fields (we freeze: alphabetical for Kahf primes, then `a2,a4,a6,bits`)
- Whitespace, casing, or punctuation

## 5. Candidate Generator (frozen)

For a non-negative integer counter `c`:

```text
suffix(c)  =  ":counter=" ‖ ascii_decimal(c)            // no leading zeros

block0(c)  =  SHA-512( master_seed ‖ ":block=0" ‖ suffix(c) )
block1(c)  =  SHA-512( master_seed ‖ ":block=1" ‖ suffix(c) )

raw(c)     =  ( block0(c) ‖ block1(c) )[: 66]            // first 66 bytes = 528 bits

val(c)     =  big_endian_int( raw(c) )
val(c)    &=  (1 << 521) − 1                             // mask to 521 bits
val(c)    |=  (1 << 520)                                 // force MSB → exactly 521-bit
val(c)    |=  1                                          // force odd

candidate(c)  =  val(c)
```

Two SHA-512 blocks are required because a single SHA-512 produces only 512
bits, but we need 521. Block indices `:block=0;` and `:block=1;` provide
unambiguous domain separation.

See `bcs521-v2-search/kahf_seeded_search.py::candidate`. Smoke test

## 6. Search Predicate

A candidate `p = candidate(c)` is accepted iff **all** the following hold:

| # | Predicate | How verified |
|---|---|---|
| 1 | `p` is prime | BPSW + 20 random Miller–Rabin rounds |
| 2 | `p` ≡ 1 mod 2 (odd) | by construction |
| 3 | `p.bit_length() == 521` | by construction |
| 4 | `n := #E(F_p)` is prime | PARI/GP `ellap` (SEA) + BPSW + MR |
| 5 | `n ≠ p` (not anomalous) | direct compare |
| 6 | `G = (0, 2)` is on `E` | identity (`4 ≡ 4`), holds for any `p > 4` |

Cofactor `h = #E / n_max_prime = 1` since `n` itself is prime.

The **winning counter** is the smallest non-negative integer `c*` for
which all predicates hold. The published prime is then
`p* := candidate(c*)`.

### Audit trail

The search tool writes a checkpoint every 30 seconds and a final
certificate JSON containing:

- The exact `canonical_input`, `master_seed`, `winning_counter`
- `p`, `n`, `a_p`, bit lengths
- Hasse-bound assertion, anomalous-check assertion
- Pending external proof items (PARI APR-CL, Sage `proof=True`,
  embedding degree, twist factorization)

Anyone can re-run `python3 kahf_seeded_search.py --verify <c*>` and
confirm the same `p` is produced.

## 7. Expected Search Cost

521-bit prime density ≈ 1 / (521 · ln 2) ≈ **1 / 361**.

So `p` is prime roughly every 361 counters on average. Among those, the
probability that `n = #E(F_p)` is *also* prime is approximately the same
density (Hasse–Mihailescu heuristic ≈ 1 / 361). So the expected number of
counters scanned is:

```text
E[c*]  ≈  361 × 361  ≈  130 000
```

At ~10 ms per BPSW check on `p` and ~75 s per SEA cardinality (Codespaces
4-core measurement on the existing V1 prime), the wall-clock estimate is:

```text
fast filter (BPSW on p)   :  130 000 × 10 ms     ≈   22 min
SEA on prime p            :     361 × 75 s       ≈  7.5 hr
TOTAL (single core)       :                      ≈  ~8 hr expected
TOTAL (4-core parallel)   :                      ≈  ~2 hr expected
TOTAL (32-core GCP)       :                      ≈  ~15 min expected
```

These are *expected* values; actual runtime can vary by 2–3× either way.
